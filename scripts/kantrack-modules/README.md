# scripts/kantrack-modules/

Every feature module of the KanTrack application. Each file has one responsibility.

---

## TypeScript core (typed infrastructure)

| File            | What it does                                                                                                                                                                                                                                                                                               |
| --------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `types.ts`      | Shared domain types: `Task`, `Column`, `Priority`, `Tag`, `NotebookItem` (incl. `expanded?: boolean`), `Clock`, `UndoEntry`                                                                                                                                                                                |
| `store.ts`      | Minimal flux-like store — `dispatch()`, `subscribe()`, `getState()`, typed reducer, `TASK_*` action types                                                                                                                                                                                                  |
| `router.ts`     | Event delegation router — `registerAction()` maps `data-action` names to handlers; `initRouter()` attaches one document-level click listener                                                                                                                                                               |
| `repository.ts` | **All** persistence logic: `getAllTasks()`, `saveTasks()`, `getAllNotebookItems()`, etc. Writes localStorage synchronously and IDB async (300 ms debounced). `flushPendingIDBWrites()` cancels any pending timer and fires the write immediately — called on `beforeunload` and before `location.reload()` |

## Key data flows

### Creating a task

`addNote()` (tasks.js) → `state.pushToNotesData()` → `saveNotesToLocalStorage()` → `repository.saveTasks()` → localStorage (sync) + IDB (async, 300 ms debounced)

On `beforeunload`, `flushPendingIDBWrites()` fires any pending debounced IDB write immediately so no data is lost if the tab is closed mid-debounce.

### Saving modal changes

Both paths share `_persistModalChanges()` which handles title, priority, tags, timer, and note content:

- **Save & Close:** `saveAndCloseModal()` → `_persistModalChanges()` → `saveNotesToLocalStorage()` → `updateNoteCardDisplay()` → `closeTaskModal()`
- **Save (keep open):** `saveModal()` → `_persistModalChanges()` → `saveNotesToLocalStorage()` → `updateNoteCardDisplay()` → resets tracking state + clears editor + re-renders history in place

### Loading on startup

`loadNotesFromLocalStorage()` (storage.js) → `repository.getAllTasks()` → localStorage (primary, always written synchronously) or IDB (fallback) → `state.setNotesData()` → render all cards

## Notable bug fixes

### Tag filter buttons did nothing after a page reload (`search.js` + `tags.js`)

After a full browser reload, clicking a tag filter button had no visible effect — no tasks were filtered and the button never turned active.

**Root cause — duplicate event listeners**

The bootstrap sequence ran two listener-attachment paths over the same DOM nodes:

1. `renderTagFilterButtons()` — rendered the buttons and attached a dynamic-import listener to each: `import('./search.js').then(m => m.toggleTagFilter(tag))`
2. `initSearch()` — queried those same buttons and attached a second, synchronous listener

On **first load** (no stored data) `renderTagFilterButtons()` showed the "no tags" message, so `initSearch()` found no buttons. Tags added later in the same session only ever got the single dynamic-import listener. On **reload** (data already in storage) both ran in sequence, and each button ended up with two listeners.

On click, the synchronous listener fired and added the tag to `currentTagFilter`; then the dynamic-import `.then()` ran as a microtask (before any repaint) and toggled the same tag back off. The user always saw only the final state — no filter applied.

**Fix**

- `initSearch()` — replaced the per-button `querySelectorAll` loop with a single delegated listener on the `#tagFilters` container. The container is permanent; its buttons are replaced dynamically. Delegation handles any child button regardless of when it was created.
- `renderTagFilterButtons()` — removed the per-button listener attachment block entirely. Event wiring is now exclusively the responsibility of `initSearch()`.

### PDF export silently dropped images on macOS (`export.js`)

When exporting a task to PDF on macOS, the resulting file contained no images even though the same workflow worked correctly on Windows.

**Root cause — two compounding bugs**

1. **Shallow `childNodes` traversal** — `exportTaskAsPDF` iterated only the direct children of the temporary DOM element. On macOS, Chrome's `contenteditable` wraps pasted content in `<div>` block elements, so `<img>` nodes were nested one level deeper and never reached.

2. **Hardcoded `'PNG'` format in `doc.addImage`** — macOS clipboard (Preview, Keynote, etc.) puts `image/tiff` data URLs on the clipboard. Passing a TIFF data URL with the format forced to `'PNG'` causes jsPDF to fail silently inside its own `try/catch`.

**Fix**

Two private helpers were added before `exportTaskAsPDF`:

- `_normalizeImageToPng(dataUrl)` — converts any image format to PNG via `<canvas>` before passing to jsPDF; PNG and JPEG pass through unchanged.
- `_walkNodesForPDF(nodes, doc, taskId, margin, maxWidth, yPosRef)` — recursively walks a `NodeList`, handling text nodes, `<img>` elements (IDB lookup → `data-src` fallback → `src` fallback, all normalised through `_normalizeImageToPng`), and any other element by descending into its `childNodes`.

The flat `childNodes` loop in `exportTaskAsPDF` was replaced with a call to `_walkNodesForPDF`. Only `exportTaskAsPDF` was affected — the JSON/encrypted export paths use a separate `querySelectorAll('img')` (recursive) approach and were never broken.
